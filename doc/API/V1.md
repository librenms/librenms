# API v1

API v1 is a modern REST API built with [Laravel Restify](https://restify.binarcode.com/). It follows the [JSON:API](https://jsonapi.org/) specification with built-in pagination, filtering, and search.

API v1 runs alongside the existing [v0 API](index.md). The v0 API continues to work as before.

## Interactive reference

Each LibreNMS instance serves its own OpenAPI 3 document and an interactive
Swagger UI:

- `GET /api/v1/openapi.json` the OpenAPI 3 specification
- `GET /api/v1/docs` Swagger UI loaded against that spec

The spec is generated from the running configuration by introspecting the
registered Restify repositories, so it always reflects what your server
actually exposes. no regeneration step is required when endpoints, fields,
filters, sorts, or relationships change. Responses are cached for 5 minutes;
append `?fresh=1` to bypass the cache during development.

To dump the same spec from the CLI (for codegen or CI):

```bash
lnms api:openapi --output=openapi.json
lnms api:openapi --validate    # syntactic validation, exits non-zero on errors
```

## Authentication

API v1 uses [Laravel Sanctum](https://laravel.com/docs/sanctum) Bearer tokens.

### Managing tokens

Generate, list, and revoke tokens via the `lnms` CLI:

```bash
# Create a token for a user
lnms api:token create admin
lnms api:token create admin --token-name="monitoring-script"

# List all tokens for a user
lnms api:token list admin

# Revoke a token by ID (use list to find the ID)
lnms api:token revoke admin --token-id=3
```

Save the output of `create` the plain-text token is only shown once.

### Using the token

Pass the token in the `Authorization` header:

```bash
curl -H 'Accept: application/vnd.api+json' \
     -H 'Authorization: Bearer YOUR_TOKEN_HERE' \
     https://librenms.example.com/api/v1/devices
```

For write requests (POST, PUT, PATCH), set the `Content-Type` header:

```bash
curl -X POST \
     -H 'Content-Type: application/vnd.api+json' \
     -H 'Authorization: Bearer YOUR_TOKEN_HERE' \
     -d '{"name": "new-group", "type": "static"}' \
     https://librenms.example.com/api/v1/device-groups
```

All responses use `Content-Type: application/vnd.api+json`. The server also accepts `application/json` as `Content-Type` for convenience. Requests with a body that use any other content type will receive `415 Unsupported Media Type`.

Unauthenticated or invalid requests return HTTP 401:

```json
{"message": "Unauthenticated."}
```

## Endpoints

For the full list of endpoints, fields, filters, and relationships, see the
interactive Swagger UI at `/api/v1/docs` or the OpenAPI document at
`/api/v1/openapi.json` on your LibreNMS instance.

## Pagination

All list endpoints are paginated. Control pagination with query parameters:

```
GET /api/v1/devices?page=2&perPage=25
```

Default page size is 15.

## Search

Use the `search` query parameter to search across the resource's searchable fields:

```
GET /api/v1/devices?search=router
```

This searches across `hostname`, `sysName`, and `os` for devices.

## Access Control

### Authorization flow

Every request runs through the same four stages. It must clear **all** of them or it is
rejected with the status shown:

```
  Request + Bearer token
          тФВ
  1. Authentication ............... valid Sanctum token?            no тЖТ 401 Unauthenticated
          тФВ yes (token тЖТ user)
  2. API access permission ........ user has `api.access`?          no тЖТ 403 Forbidden
          тФВ yes
  3. Per-action authorization ..... policy allows this verb?        no тЖТ 403 Forbidden
          тФВ yes                       (viewAny / view / create / update / delete)
  4. Result scoping ............... record(s) within user's scope?  out of scope тЖТ filtered
          тФВ                                                          (a single show тЖТ 404)
          тЦ╝
     Response
```

1. **Authentication** the `auth:sanctum` middleware resolves the token to a user. See [Authentication](#authentication).
2. **API access permission** the user must hold the `api.access` permission. It is granted to the **admin** and **global-read** roles out of the box; other roles need it added explicitly. Without it, no v1 endpoint is reachable.
3. **Per-action authorization** each HTTP verb maps to a Laravel policy method on the resource's model: `index тЖТ viewAny`, `show тЖТ view`, `POST тЖТ create`, `PATCH/PUT тЖТ update`, `DELETE тЖТ delete` (Restify also checks `allowRestify`). Two shortcuts apply: **admins** pass every check, and **global-read** users pass every *read* check (`view*` / `show`).
4. **Result scoping** even when a user may use an endpoint, the rows they get back are filtered to what they're allowed to see. How that filter works depends on the repository's *flavor* (below). A `show` for an out-of-scope id returns `404`.

### Roles at a glance

| Role | API access | Can read | Can write (create/update/delete) |
|------|:---:|---|---|
| **admin** | тЬУ | everything | everything |
| **global-read** | тЬУ | everything (read-only) | no (unless a policy allows it) |
| **user** / custom | only if `api.access` is granted | only resources they're scoped to | only where a policy allows |

### Repository scoping flavors

Stage 4 is implemented per repository. Most resources fall into one of these flavors:

| Flavor | How results are filtered | Example resources |
|--------|--------------------------|-------------------|
| **Device-scoped** | To devices the user has access to. | `devices`, `sensors`, `processors`, `storage`, `mempools`, `alerts`, `alert-rules`, `device-groups`, `locations`, `bills`, `inventory`, BGP/OSPF/MPLS peers & tunnels, тАж (~44 resources) |
| **Port-scoped** | To ports the user is granted **plus** every port on a device they can see (broader than device access). | `ports`, `port-statistics`, `transceivers`, `ipv4-addresses`, `ipv6-addresses`, `routes`, `port-vlans`, `pseudowires`, тАж (~19 resources) |
| **Global** | Not filtered per record visible to anyone who clears stages 1тАУ3 (typically global-read or admin). | `alert-templates`, `poller-groups`, `poller-clusters`, `ipv4-networks`, `ipv6-networks`, `auth-logs`, тАж |
| **Owner-scoped** | To the caller's own records. | `users` (admins see all; everyone else sees only their own profile) |

For how a new repository opts into a flavor (the `DeviceScopedRepository` / `PortScopedRepository`
traits and the `DeviceRelatedModel` / `PortRelatedModel` base models), see
[Creating API v1 Resources тА║ Access Control Patterns](../Developing/Creating-API-Resources.md#access-control-patterns).

### Per-resource notes

- **Devices and Ports** Global-read users see all; others see only the devices/ports they've been granted.
- **Alert Templates** Global-read users can view; only admins can create, update, or delete.
- **Alert Rules** Viewable with global read; only admins can create, update, or delete. Results are scoped to devices the user has access to.
- **Device Groups** Viewable with global read; only admins can create, update, or delete. Results are scoped to the groups the user can access.
- **Users** Only admins can list all users; non-admin users can only view their own profile.
